# Publishing vs. Raw: Entity Strategy in Strapi

This document analyzes the "Draft & Publish" configuration across the Daicer backend. It categorizes entities into "Content" (Curated) vs. "Data" (Runtime) and identifies misconfigurations that may be causing state synchronization issues.

## The Core Philosophy

To ensure data integrity and proper game flow, we must strictly distinguish between two types of entities:

1.  **Curated Content (Draft & Publish: ENABLED)**

    - **What**: Game rules, static assets, lore, templates.
    - **Why**: Creators need to "Draft" changes (e.g., balance a Monster, write a Spell) without breaking the live game. Changes should only appear when explicitly "Published".
    - **Behavior**: API queries return the `published` version by default. Drafts are invisible to players.

2.  **Runtime Data (Draft & Publish: DISABLED)**
    - **What**: Game state, chat logs, player HP, room positions, event streams.
    - **Why**: These change instantly and frequently. There is no concept of "Drafting" damage. If a player takes 5 damage, it is a fact, not a proposal.
    - **Risk**: If enabled, an update (e.g., `-5 HP`) might save as a Draft. The Read API (fetching `published`) will return the old HP, effectively "reverting" the game state.
